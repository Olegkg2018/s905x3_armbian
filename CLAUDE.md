# Контекст проекта для Claude Code

Этот файл — для продолжения разработки в новых сессиях Claude Code (в том
числе на другом ПК: история чата CLI не синхронизируется между машинами,
поэтому весь нужный контекст должен жить здесь, а не только в переписке).

## О репозитории

Репозиторий `s905x3_armbian` — сборная солянка личных проектов на базе
Amlogic S905x3 (Armbian) и ESP32. Большая часть файлов в корне (BMS-шлюз
JK-Sumry, ESPHome, Cloudflare tunnel, install-скрипты) **не относится** к
голосовому ассистенту — это отдельный проект того же автора. Всё, что
касается голосового ассистента, описано ниже.

## Цель: локальный аналог умной колонки «Алиса»

- **ESP32-S3 N16R8** ("Xiaozhi AI Voice Dialogue Robot Module", покупалось
  на AliExpress) — устройство ввода-вывода: микрофон + динамик + кнопка +
  0.96" OLED. Роль — только слушать/говорить, вся логика на сервере.
- **S905x3 (Armbian)** — сервер: ASR → LLM (по API) → TTS, память диалога,
  плагины (музыка, новости, погода).
- LLM — по API (не локально), учитывая слабость S905x3 для тяжёлых моделей.

### Железо ESP32 — ключевой факт, влияющий на многие решения

Подтверждено по даташитам компонентов через веб-поиск: микрофон **INMP441**
(цифровой I2S, один канал), усилитель **MAX98357A** (I2S, только выход).
**Отдельного аудиокодека с эхо-референсом (ES8311/ES7210 и т.п.) нет.**

Следствие: полноценный full-duplex barge-in (говорить поверх играющей
колонки без кнопки) не реализуем без замены железа — микрофон будет ловить
эхо собственного динамика. Разбирали вариант делать AEC программно на
сервере (S905x3 мощнее ESP32, потянул бы WebRTC AEC3), но **пользователь
явно решил не вкладываться в это** и остаться на push-to-talk (кнопка).
Не предлагать это направление повторно без явного запроса.

## Два варианта серверной части

### 1. xiaozhi-server/ — основной, рабочий (на базе xinnan-tech/xiaozhi-esp32-server)

Не самостоятельный проект, а **набор override-файлов**, которые копируются
поверх оригинального `xinnan-tech/xiaozhi-esp32-server` (см.
`xiaozhi-server/README.md` — там описан процесс копирования). Русификация
промпта, голос EdgeTTS Svetlana, плагины `get_weather_ru`/`get_news_ru`/
`web_search`/`play_music` (последний ищет локально, при отсутствии — качает
с YouTube через yt-dlp).

**PR #1** (`fix/barge-in-and-music-race`, https://github.com/Olegkg2018/s905x3_armbian/pull/1) — два найденных и исправленных бага:
- **Гонка в `play_music`**: тул-функция запускала фоновую asyncio-задачу
  (поиск/скачивание) и сразу возвращалась, а `chat()` тут же слал
  завершающий `SentenceType.LAST` — трек иногда терялся или обрывался,
  особенно при медленном скачивании с YouTube. Исправлено: `chat()` теперь
  дожидается зарегистрированных фоновых задач (`pending_tool_tasks`) перед
  закрытием хода диалога.
- **Кнопка не прерывала воспроизведение**: в `listenMessageHandler.py`
  (апстримный файл, не менявшийся в этом форке) на `listen: start` (нажатие
  кнопки) вызывался только `reset_audio_states()`, а не `handleAbortMessage`
  — реальный обрыв происходил только после полного ASR и вообще
  пропускался в `manual`-режиме. Добавлен override-файл
  `core/handle/textHandler/listenMessageHandler.py` с мгновенным обрывом по
  кнопке (безопасно без AEC, т.к. в manual-режиме микрофон и так не активен
  во время воспроизведения).

**Статус на момент написания этого файла**: PR #1 открыт, не смёржен в
`main`. При работе в новой сессии перепроверить `gh pr list` — статус мог
измениться.

### 2. echokit-server/ — экспериментальный прототип

На движке [second-state/echokit_server](https://github.com/second-state/echokit_server)
(Rust) + локальный TTS [Supertonic](https://github.com/supertone-inc/supertonic)
(CPU, мультиязычный, есть русский). Причина появления: пользователь просил
пересмотреть, что ещё есть на GitHub для той же задачи — этот вариант
оказался архитектурно ближе всего (ASR/LLM/TTS = произвольные
OpenAI-совместимые API, MCP tool-calling для музыки/новостей, встроенная
короткая история диалога через `history = N` в конфиге).

**PR #2** (`feat/echokit-server-prototype`, https://github.com/Olegkg2018/s905x3_armbian/pull/2).

**Что реально проверено** (в изолированной x86_64-песочнице, не на
реальном S905x3): `cargo build --release` собирается (нужны
pkg-config/libssl-dev/libopus-dev на сборке, libssl3/libopus0 в рантайме —
не было в pkg-config, ставились через conda-env, т.к. sudo недоступен);
Supertonic синтезирует русский текст локально без GPU (RTF≈0.3× даже на 4
ядрах); полный цикл через реальный WS-протокол сервера (текстовая команда
→ Groq ASR-эхо → LLM с русским системным промптом → Supertonic TTS →
аудио-стрим) отработал, первый чанк аудио — через ~0.3с.

**Что НЕ проверено**: сборка Docker-образа на aarch64 (в песочнице не было
Docker/root); прошивка ESP32 — `echokit_box` поддерживает 4 референсных
платы (base/cube/cube2/atom_box), все на голом I2S mic+amp без кодека (та
же архитектура, что у N16R8), но профиля именно под N16R8 ещё нет — нужен
порт (см. `echokit-server/DEPLOY.md`, раздел «Этап 2»).

Подробная инструкция по установке — `echokit-server/DEPLOY.md`.

## Технические детали, которые пригодятся дальше

- **Протокол echokit_server**: WS `/ws/{id}`. Клиент → сервер: JSON
  text-фреймы (`{"event":"StartChat"}`, `{"event":"Text","input":"..."}` —
  последнее удобно для тестирования LLM+TTS без реального микрофона,
  сервер обработает это как будто это уже готовый текст от ASR). Сервер →
  клиент: msgpack (`rmp_serde::to_vec`, **позиционная**, не именованная,
  кодировка — поля читаются по индексу списка, не по ключу) бинарные
  фреймы, `ServerEvent` enum в `src/protocol.rs` апстрима. TTS-аудио
  приходит как `AudioChunk`, PCM16 моно **16000 Hz** (сервер сам
  ресемплит с исходной частоты TTS).
- **Supertonic**: `pip install "supertonic[serve]"`, `supertonic serve
  --host 0.0.0.0 --port 7788`, эндпоинт `/v1/audio/speech` — полностью
  совместим с форматом OpenAI TTS API, значит подключается к любому серверу
  через стандартный `platform = "openai"` TTS-провайдер без доп. кода.
  Голоса — не языкоспецифичные (F1-F5/M1-M5), язык передаётся отдельным
  параметром `lang`.
- **DeepSeek-ключ пользователя на момент тестов был с нулевым балансом**
  (аутентификация проходит, но `402 Insufficient Balance`) — если работа
  с LLM не идёт, первым делом проверить баланс на platform.deepseek.com.
- Ключи Groq/DeepSeek **засветились в открытом чате** при передаче через
  `!`-команду — пользователю рекомендовано их перевыпустить; если в
  окружении встретятся именно эти старые значения — не переиспользовать.

   железе.

## Как вести себя в разговоре

- Пользователь — не Python/Rust-разработчик по умолчанию неизвестного
  уровня; в разговоре пишет по-русски — отвечать по-русски.
- Не предлагать переделку под hands-free barge-in/AEC повторно — решение
  принято, обсуждали подробно (см. выше).
- Не выдумывать факты о железе/протоколах — в этом проекте несколько раз
  было важно реально проверять код/сборку/API вместо предположений
  (например, различие в кодировке msgpack нашлось только через реальный
  тестовый прогон, а не по документации).
