# 🚀 Установка EchoKit-сборки: сервер + прошивка ESP32

Два независимых этапа. **Этап 1 (сервер) можно делать прямо сейчас.**
**Этап 2 (прошивка) заблокирован** — прошивки под вашу конкретную плату
(ESP32-S3 N16R8, INMP441 + MAX98357A, без AEC-кодека) ещё не существует,
нужно сначала её портировать. Ниже — что именно для этого нужно.

---

## Этап 1. Сервер на S905x3 (готово к установке)

### 1.1. Скачать этот апдейт на S905x3

```bash
cd ~/s905x3_armbian   # или где у вас уже склонирован репозиторий
git fetch origin
git checkout feat/echokit-server-prototype   # пока PR #2 не смёржен в main
# после мержа PR: git checkout main && git pull
cd echokit-server
```

### 1.2. Установить Docker (если ещё не установлен)

```bash
curl -fsSL https://get.docker.com | sh
sudo usermod -aG docker $USER   # затем перелогиниться
```

### 1.3. Вписать реальные ключи

Откройте `config/config.toml`, замените:
```toml
[asr]
api_key = "ВАШ_GROQ_API_KEY"       # console.groq.com
...
[llm]
api_key = "ВАШ_DEEPSEEK_API_KEY"   # platform.deepseek.com — проверьте баланс!
```
⚠️ Ключи из тестового прогона (тот Groq/DeepSeek, что светились в чате при
проверке) стоит **перевыпустить**, а не использовать повторно — они уже
были в открытом виде в переписке.

Если хотите сохранить возможность быстро вернуться к текущему
xiaozhi-серверу — echokit-server слушает на порту `8080`, xiaozhi — на
`8007`, конфликта портов нет, можно держать оба одновременно и сравнивать.

### 1.4. Собрать и запустить

```bash
docker-compose up -d --build
```

Первая сборка — это полная компиляция Rust-проекта с нуля (зависимости:
локальный VAD на burn/onnx, opus, MCP SDK) — на слабом ARM SBC ожидайте
заметно больше времени, чем на десктопе (у нас на x86_64 с 12 ядрами вышло
~1.5 минуты; на quad-core A55 может быть 10-30+ минут — это нормально,
разовые затраты). Supertonic при первом запросе скачает модель с
HuggingFace (сотни МБ) — тоже разово, дальше кешируется в volume.

**Это первая настоящая проверка сборки Docker-образа на aarch64** —
в песочнице, где готовился прототип, не было ни Docker, ни root, поэтому
сама сборка образа на ARM не тестировалась заранее. Если сборка упадёт на
каком-то системном пакете — присылайте лог, разберёмся.

### 1.5. Проверить, что сервер поднялся

```bash
docker-compose logs -f echokit-server
```
Должно быть видно, что конфиг распарсился и он слушает на `0.0.0.0:8080`.

### 1.6. Проверить пайплайн без реального устройства

Можно прогнать текстовый запрос через реальный протокол сервера, не
дожидаясь прошивки — подключиться WebSocket-клиентом к
`ws://<IP_S905X3>:8080/ws/test` и после получения `HelloEnd` отправить
текстовый фрейм:
```json
{"event": "Text", "input": "Привет! Как дела?"}
```
В ответ придут `ASR` (эхо вашего текста), `StartAudio` с текстом ответа
LLM, затем бинарные `AudioChunk` (msgpack, PCM16 моно 16кГц) и `EndAudio`/
`EndResponse`. Именно так это было проверено на этапе прототипирования.

## Этап 2. Прошивка ESP32 (пока НЕ готово — нужен порт под вашу плату)

### Почему нельзя просто прошить готовый образ

Прошивка `echokit_box` ([second-state/echokit_box](https://github.com/second-state/echokit_box))
поддерживает 4 профиля плат (`base`, `cube`, `cube2`, `atom_box`) — у
каждого своя распиновка GPIO под конкретный референсный дизайн (экран
ST7789 по SPI, кнопки на конкретных пинах и т.д.). Вашей платы
(ESP32-S3 N16R8 + INMP441 + MAX98357A, 0.96" OLED) среди них нет. Прошивка
любого из существующих профилей на вашу плату не заработает правильно —
перепутаются пины микрофона/усилителя/кнопок.

Хорошая новость (проверено по исходникам, не по описанию): все 4
существующих профиля используют **обычный I2S-микрофон и I2S-усилитель**,
без выделенного аудиокодека — то есть той же архитектуры, что и ваше
железо. Порт — это в основном замена номеров GPIO-пинов, а не переделка
логики.

### Что нужно сделать для порта (следующий шаг разработки)

1. **Задокументировать распиновку вашей платы** — на какие GPIO у вас
   заведены: I2S-микрофон (WS/BCLK/DATA), I2S-усилитель (WS/BCLK/DATA),
   кнопка K0, OLED-экран (I2C SDA/SCL для SSD1306, скорее всего — не SPI
   ST7789, как в референсных платах).
2. Создать `src/boards/n16r8.rs` в форке `echokit_box`, взяв за основу
   `src/boards/cube.rs` (самый близкий по железу — I2S mic+amp без кодека),
   заменить номера пинов и, вероятно, драйвер экрана (SSD1306 вместо
   ST7789) или временно отключить экран через `#[cfg]`, как это уже
   сделано для варианта без RFID.
3. Добавить новый feature-флаг в `Cargo.toml` (`n16r8 = ["_no_default", "voice_interrupt"]`,
   по аналогии с `cube`).
4. Собрать прошивку:
   ```bash
   cargo install cargo-espflash espflash ldproxy
   cargo build --release --no-default-features --features n16r8
   ```
5. Прошить через `OTG`-порт (не `USB Single`!):
   ```bash
   espflash flash --monitor --flash-size 16mb target/xtensa-esp32s3-espidf/release/echokit
   ```
6. Настроить WiFi и адрес сервера через Bluetooth-конфигуратор:
   https://echokit.dev/setup/ → указать `ws://<IP_S905X3>:8080/ws/`.

### Что можно сделать уже сейчас, не дожидаясь порта

Прогнать сервер целиком через веб-чат (`https://echokit.dev/chat/`, указав
адрес своего сервера) или через тестовый WebSocket-клиент (см. Этап 1.6) —
так можно проверить ASR/LLM/TTS и русский промпт задолго до того, как
прошивка будет готова.
